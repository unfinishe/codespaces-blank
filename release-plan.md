# Google Play release plan for Android Pic Sort

This checklist is a repository-specific publication plan for releasing `de.thomba.andropicsort` on Google Play.

Use it together with:
- `README.md`
- `PLAY_STORE_LISTING.md`
- `PRIVACY_POLICY.md`
- `RELEASE_NOTES.md`
- `app/build.gradle.kts`
- `app/src/main/AndroidManifest.xml`

## 1. Immediate blockers to resolve before submission

### 1.1 Align version numbers everywhere
- [x] Choose the release version to publish first.
- [x] Update `gradle.properties`:
  - `app.versionCode`
  - `app.versionName`
- [x] Make the same version consistent in:
  - `RELEASE_NOTES.md`
  - `PLAY_STORE_LISTING.md`
  - any Play Console release notes draft

**Current repo finding**
- `gradle.properties` is aligned to `app.versionCode=5` and `app.versionName=1.0.5`.
- `RELEASE_NOTES.md` is aligned to version `1.0.5` (heading and footer).
- `PLAY_STORE_LISTING.md` is aligned to version `1.0.5`.

**Hint**
- Play Console requires every new upload to have a strictly higher `versionCode` than the previously uploaded artifact.
- Fix the mismatch first, otherwise store text, release notes, and the binary will contradict each other.

### 1.2 Fix privacy and data safety wording to match the implementation
- [x] Review `PRIVACY_POLICY.md` and `PLAY_STORE_LISTING.md` against the actual app behavior.
- [x] Update all wording that currently implies the app stores nothing between sessions.
- [x] Explicitly document that the app stores local preferences and folder URIs on device.
- [x] Review the Android backup statement for correctness.

**Current repo finding**
- `UiSettingsStorage.kt` stores settings in Preferences DataStore.
- `MainActivity.kt` and `MainViewModel.kt` persist SAF tree access and restore folder selections across sessions.
- `PRIVACY_POLICY.md` now documents local on-device settings persistence consistently.
- `PRIVACY_POLICY.md` now aligns backup wording with `android:allowBackup="false"`.

**Hint**
- Play Data safety answers must match the shipped APK/AAB and the privacy policy.
- Local on-device preference storage still counts as handled data for policy wording, even if nothing is sent to a server.

### 1.3 Replace outdated permission wording with SAF-based wording
- [x] Remove or rewrite statements that say the app requests `READ_EXTERNAL_STORAGE` or `WRITE_EXTERNAL_STORAGE`.
- [x] Describe storage access as user-granted folder access through the Storage Access Framework.

**Current repo finding**
- `app/src/main/AndroidManifest.xml` declares no storage runtime permissions.
- The app uses folder picking and persisted URI permissions instead.
- `PLAY_STORE_LISTING.md`, `PRIVACY_POLICY.md`, and `RELEASE_NOTES.md` use SAF-based access wording and no longer mention legacy storage permissions.

**Hint**
- For Play review and user trust, describe the real access model: user-selected folders only, via SAF.

### 1.4 Prepare a real release build path
- [x] Decide whether you will publish from Android Studio or CI.
- [x] Add a release signing strategy:
  - local keystore for the upload key, or
  - CI secret-based signing
- [x] Configure `signingConfigs` and release signing if you want reproducible local release builds.
- [x] Add a documented release command for producing the Play artifact.
- [x] Prefer Android App Bundle (`.aab`) for Play uploads.

**Current repo finding**
- Architecture decision accepted in `arc42/09_architecture_decisions.md` (ADR-008): local-first release process without CI for now.
- Local release signing is defined via `keystore.properties`-based configuration in `app/build.gradle.kts`.
- Local release command and artifact capture are documented via `./scripts/release-local.sh` and `docs/local-release-play-upload.md`.
- `README.md` now references the local-first release workflow.
- `isMinifyEnabled` is currently `false`.

**Hint**
- A debug APK is not a Play Store release artifact.
- Before first submission, verify that you can produce a release bundle successfully.

### 1.5 Raise confidence with app-level testing evidence
- [x] Run all existing core tests and keep the results.
- [x] Add at least a small set of app-side tests before release:
  - `app/src/test/` has `SortProgressTest` (6 tests) and `MainViewModelTest` (15 tests)
  - ViewModel tests run on JVM via Robolectric – no emulator required
- [ ] One manual or automated regression pass for dry run vs copy/move behavior (see section 6.2)
- [x] Create a short test report for your own release archive.
  - `test-reports/v1.0.5-20260415.md` committed (auto-generated by `scripts/release-local.sh` on future releases)

**Current repo finding**
- `core/src/test/kotlin/...`: **38 passing tests** across 6 test classes.
  - `SupportedImageFormatsTest` – 8 tests incl. case-insensitivity, null, blank
  - `YearMonthFolderSchemaTest` – 7 tests incl. all 12 months, zero-padding, uppercase
  - `InputFilePolicyTest` – 7 tests incl. null/blank filename paths
  - `SortReportTest` – 4 tests for counter fields, dry run flag, error buckets
  - `ConflictResolverTest` – 6 tests
  - `AppLocalePolicyTest` – 6 tests
- `app/src/test/java/.../sort/SortProgressTest` – 6 passing tests (pure-Kotlin data class)
- `app/src/test/java/.../ui/MainViewModelTest` – **15 passing tests** via Robolectric 4.14.1 + SDK 35:
  - source==target guard, missing-folders guard, double-start guard
  - dry run, COPY/MOVE mode, RENAME/OVERWRITE policy, METADATA_THEN_FILE/FILE_ONLY date mode
  - successful sort produces report, inaccessible URI dropped on restore
- `MainViewModel` uses injected `ioDispatcher` (default `Dispatchers.IO`); tests inject `StandardTestDispatcher` for deterministic coroutine control.
- **Note:** All tests require JDK 21. Use `JAVA_HOME=.tools/jdk-21.0.10+7` if the system default is JDK 26.

**Hint**
- Play does not require automated tests, but you need enough evidence to survive review issues and user reports after launch.
- The 15 ViewModel tests cover the critical state-machine paths from section 6.2.
